# Правила документации WORK

Стандарт системы документирования платформы WORK. Описывает принципы, структуру и механику ведения документации на каждом уровне иерархии.

---

## 1. Принципы

### 1.1. Иерархичность

Навигация по документации осуществляется через структуру папок. Путь к файлу сам по себе несёт контекст. Явные ссылки на родительские или дочерние readme не нужны — иерархия директорий обеспечивает связность.

### 1.1.1. Ссылки на внутренние файлы WORK

Когда всё же нужна кликабельная ссылка на файл внутри work (не «сырой» путь для Git), использовать только WORK-формат формы:

```markdown
[`sources/core.js`](/sources/core.js/~/handlers/pages/form/)
```

Шаблон: `/абсолютный/путь/к/файлу/~/handlers/pages/form/`.

- Относительные href вида `../../sources/page.html` для навигации в UI не использовать — в форме md они резолвятся относительно URL handler’а и ломаются.
- Внешние `http(s):` / `mailto:` / якоря `#` — как обычно.
- Viewer при показе дополнительно превращает подходящие пути в бэктиках и простые markdown-ссылки в WORK-формат; при ручном написании всё равно предпочитать явный WORK-href.

### 1.2. Двухуровневость

Каждый документ отвечает одновременно для двух потребителей:

- **Разработчик** (человек или ИИ-агент) — понимает техническое устройство для развития и модификации системы.
- **ИИ-управляющий процессами** — понимает прикладное назначение для управления задачами, процессами и ресурсами организации.

Оба измерения описываются в одном документе, без дублирования.

### 1.3. Минимализм

Документация не дублирует код. `readme.md` подробно описывает устройство и возможности, но не пересказывает реализацию. `progress.md` фиксирует факт + причину, а не пересказывает изменения в коде. Если код самодокументирован — документация описывает зачем и как это работает концептуально.

### 1.4. Актуальность

Документация обновляется вместе с кодом. Устаревший документ хуже отсутствующего. Раздел «Состояние» в `readme.md` и раздел «Последние изменения» в `progress.md` — индикаторы актуальности.

### 1.5. Глобальные механизмы — в корне

Глобальные механизмы (наследование, `~`, `$class`, `$public`, реактивность) описываются один раз в корневом `readme.md`. Локальные документы не повторяют их, а опираются на корневой контекст через иерархию.

### 1.6. Изоморфность WORK

Процесс разработки в VSCode зеркалирует паттерн `file → history` платформы WORK. `progress.md` — текущее состояние (перезаписывается), `.progress.md/history/TIME.USER.md` — неизменный снимок (как `history/<date>/время.uid.ext`).

### 1.7. Машиночитаемость

Структура разделов единая и предсказуемая. Это позволяет ИИ эффективно навигировать по дереву знаний, понимать контекст любой точки и использовать инструменты системы. Отклонения от стандарта недопустимы.

### 1.8. Человекоориентированность

`readme.md` хорошо оформлен и понятен. Markdown с заголовками, списками, таблицами. Не сухой протокол, а понятный рассказ о точке системы. Документ пишется для людей, а не только для машин.

## 2. Типы документов

| Документ | Назначение | Где | Когда обновляется |
|---|---|---|---|
| `readme.md` | Описание точки системы | На каждом уровне | При изменении структуры или концепции |
| `progress.md` | История работы и решений | На каждом уровне с активной разработкой | При выполнении работы |
| `rules.md` | Стандарт документации | В корне | При изменении принципов |

### 2.1. Дополнительные документы

- `whitepaper.md` — фундаментальная архитектурная декларация. Не входит в стандарт локальной документации.
- `#system/` — системная конфигурация (не документируется через readme).

## 3. Структура `readme.md`

Каждый `readme.md` содержит 6 разделов в указанном порядке:

### 3.1. Что это

Назначение элемента. Одно-два предложения. Техническое и прикладное измерение.

### 3.2. Зачем это нужно

Роль в архитектуре и в управлении. Какую проблему решает. Зачем существует.

### 3.3. Как это работает

Принципы и ключевые механизмы. Концептуальное описание, не код. Доступные функции и инструменты — для ИИ-управляющего.

### 3.4. Из чего это состоит

Перечень содержимого: файлы, подпапки, классы, методы. Каждый элемент описан в двух измерениях — техническое имя + прикладное назначение в скобках.

Пример:

```
- `folder.js` — класс `$folder`: дерево элементов, `children`, `get_item` (управление деревом папок, навигация, доступ к дочерним элементам)
- `class.js` — класс `$class`: merge/diff, logs, metadata (наследование поведения, журнализация операций, доступ к истории)
```

### 3.5. В каком это состоянии

Честный статус. Что реализовано, что работает, что в процессе, что не работает. Метки:

- ✅ работает
- 🔧 в разработке
- 🧪 прототип / эксперимент
- ❌ не реализовано
- ⚠️ устарело

### 3.6. Дальнейшие планы

Направления развития. Открытые вопросы. Что планируется сделать.

## 4. Структура `progress.md`

```markdown
# Прогресс: <название>

## Последние изменения
- [дата] Сделано X — причина/решение
- [дата] Сделано Y

## В работе
- Текущая задача

## Ключевые решения
- Решение: выбор A вместо B. Причина: ...

## Блокеры / Открытые вопросы
- Проблема X — ждём Y
```

### 4.1. Принципы `progress.md`

1. **Локальность** — на каждом уровне свой progress. Не один глобальный лог, а иерархия логов.
2. **Хронологичность** — последние события наверху (как `git log`).
3. **Краткость** — одно-два предложения на запись. Не пересказ кода, а факт + причина.
4. **Решения важнее действий** — «почему выбрали этот подход» ценнее, чем «изменили файл X».
5. **Архив** — старые записи можно сворачивать в раздел `## Архив` внизу или удалять, перенося в git-историю.

## 5. Механика `progress.md` (mirror WORK pattern)

При каждом значимом действии ИИ-агента:

1. **Перезаписать** `progress.md` — добавить новые события наверх (текущее состояние).
2. **Сохранить копию** в `.progress.md/history/YYYY-MM-DD/TIMESTAMP.USER.md` — неизменный снимок.

Формат имени снимка: `TIMESTAMP.USER.md` внутри папки `history/YYYY-MM-DD/`

- `YYYY-MM-DD` — дата (папка по дате, как в WORK)
- `TIMESTAMP` — время в миллисекундах (как в WORK)
- `USER` — имя агента (модель ИИ или `cline` для VSCode-разработки)
- `.md` — расширение

Пример: `.progress.md/history/2026-07-19/178445411268.cline.md`

Имя снимка несёт те же атрибуты, что и в WORK: «когда · кто».

### 5.1. Структура папки

```
<папка>/
├── readme.md                                ← описание (обновляется при изменениях)
├── progress.md                              ← текущее состояние (перезаписывается)
└── .progress.md/
    └── history/
        └── 2026-07-19/
            ├── 178445411268.cline.md        ← неизменные снимки
            ├── 178445500000.cline.md
            └── ...
```

### 5.2. Git

`.progress.md/history/` можно добавить в `.gitignore` (как локальный аналог `history/` в WORK, который тоже не версионируется). Решение за пользователем.

## 6. Чек-лист для нового `readme.md`

- [ ] Раздел «Что это» — техническое + прикладное назначение
- [ ] Раздел «Зачем это нужно» — роль в архитектуре и управлении
- [ ] Раздел «Как это работает» — механизмы + доступные функции
- [ ] Раздел «Из чего это состоит» — перечень с двумя измерениями
- [ ] Раздел «В каком это состоянии» — честный статус с метками
- [ ] Раздел «Дальнейшие планы» — направления развития
- [ ] Оформление — заголовки, списки, таблицы, читаемость
