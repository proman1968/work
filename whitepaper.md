# WORK / ODANT-WORK: белая книга

Архитектурная декларация платформы: онтология, модель, ИИ, границы, эволюция.

---

## 1. Назначение

WORK — платформа для управления деятельностью организаций через файлово-событийную модель с глубокой интеграцией ИИ.

Главный пользователь системы — **организация**: иерархия подразделений, исполнителей, руководителей и процессов. В предельном случае это может быть один человек, но модель рассчитана на масштабируемую организационную иерархию.

WORK не сводится к файловому менеджеру или чат-боту. Это **целостный организм**, который:

- чувствует всё, что происходит внутри (регистрирует операции);
- видит всё, что необходимо для достижения целей снаружи (интегрируется через сенсоры);
- помнит всё, что когда-либо случилось (история + логи + журнал = память);
- развивается сам, адаптируясь под меняющиеся потребности организации (самомодификация).

## 2. Фундаментальная онтология

### 2.1. Операция — атомарный факт

**Операция** — это фундаментальная, реальная единица деятельности. Она существует в конкретной точке пространства-времени и является абсолютным, неопровержимым фактом: «в момент T участник U совершил действие D над объектом O, результат — артефакт A».

Операцию нельзя «описать неправильно» — она просто случилась. Как квантовое событие в физике, операция дискретна, измерима и необратима.

### 2.2. Процесс — проекция наблюдателя

**Процесс** — это не объект, а **взгляд с определённой точки зрения** на последовательность операций.

Если попытаться ответить, к какому процессу относится любая атомарная операция, окажется, что таких процессов можно найти ровно столько, сколько участников связано с этой операцией. Бухгалтер видит «закрытие месяца», менеджер — «сдачу проекта», директор — «выполнение KPI квартала» — и все они правы.

Поэтому процесс **нельзя смоделировать заранее** как жёсткую цепочку. Процесс можно только:

- **восстановить** пост-фактум — собрать проекцию операций для конкретного наблюдателя;
- **предсказать** вперёд — предложить следующий шаг, совпадающий с вектором цели.

Это очень напоминает схлопывание волновой функции: есть поток операций (волна), и есть конкретный наблюдатель, для которого этот поток «сворачивается» в понятную картину процессов (частица).

### 2.3. Управление = наблюдение + вектор цели

Если система детально регистрирует все операции, то:

- **прошлое** — восстанавливается из «хлебных крошек» (что было);
- **настоящее** — агрегируется в личную картину для каждого участника (что есть);
- **будущее** — планируется ИИ как следующий шаг, совпадающий с вектором цели (что делать).

## 3. Файл как универсальный квант

### 3.1. Всё есть файл

Принцип «всё есть файл» в WORK — не философский лозунг, а **инженерное решение**.

Файл — универсальный носитель любого содержания деятельности:

| Категория | Примеры |
|---|---|
| **Рабочие артефакты** | документы, презентации, таблицы, чертежи, видео, фото |
| **Операционные записи** | первичные документы, проводки, счета, накладные |
| **Коммуникации** | сообщения, письма, записи звонков, протоколы встреч |
| **События** | календарные события, задачи, назначения |
| **Результаты ИИ** | ответы, анализы, сгенерированные материалы |
| **Служебные записи** | логи, метаданные, индексы |

Унификация носителя означает унификацию обработки: один механизм регистрации, один механизм журнализации, один механизм поиска и памяти.

### 3.2. Файл = артефакт, лог = факт

Ключевое разделение:

- **Файл** — это **артефакт**: тело результата операции (договор.xlsx, message.txt, response.md).
- **Запись в `data.logs`** — это **факт**: «операция произошла, вот её время, автор, путь, связи».

Сам по себе файл может быть перезаписан или удален. Но факт операции, зафиксированный в логе, — неизменен. Лог — единственный источник правды о прошлом.

### 3.3. Принцип минимального порога

Регистрируется только то, что **имеет влияние на вектор цели**. Это автоматически отсекает шум: системные детали (открытие соединения, пересчёт кэша) не важны и не логируются. Важно лишь то, что меняет наблюдаемое состояние организации в направлении цели.

## 4. Регистрация операций

### 4.1. Единый механизм

Любое действие, меняющее состояние организации, проходит один и тот же путь:

1. Пользователь, ИИ, handler или внешний сенсор инициирует действие.
2. Сохраняется файл-артефакт (через `save_file` или `save_files`).
3. классе раскладывает файл по назначению внутри метапапки (text, image, logs, js, etc).
4. Создаётся копия в `history/<date>/время.uid.ext` — неизменный снимок.
5. Создаётся запись в `data.logs` — факт операции с метаданными.
6. Запись попадает в ленту событий, RAG-индексацию и файловые обработчики.

### 4.2. Тройная журнализация

Каждая операция оставляет след одновременно в **трёх** логах:

1. **Личный лог исполнителя** (`/users//{uid}/$user`) — «что я делал».
2. **Лог рабочей точки** (storage, где выполнено действие) — «что происходило в этом подразделении/проекте».
3. **Логи контрагентов** (получатели/отправники) — «кто с кем взаимодействовал».

Это уже частично реализовано: `save_to_log` через `_writeLogTo` пишет в `$class` и в `authorCabinet`, а `receivers` добавляет адресатов. Принцип: операция → fan-out во все затронутые логи.

### 4.3. Структура записи лога

Запись `data.logs` содержит:

- `time` — временная метка;
- `sender` — автор операции;
- `path` — путь к историческому снимку артефакта;
- `ext` — тип/расширение файла;
- `content` — содержимое для сообщений и ответов;
- `includes` — связи с другими операциями (цепочки);
- `receivers` — получатели (для коммуникаций);
- `mainContext` — стартовый контекст (для порождённых ИИ операций).

Имя history-файла `время.uid.ext` само по себе несёт тройку «когда · кто · что».

### 4.4. Эволюция типологии

Сегодня есть базовые типы операций: `message.txt`, `task.ai`, `response.md`, `outbox.eml`, `files.pack`, `skill.skill`.

Типология наращивается **эволюционно по мере надобности**: завтра появятся `invoice.json`, `assignment.op`, `permission.op`, `call.record` и т.д. При этом базовый механизм один: операция → файл → лог. Новые классы добавляются как новые `$class`-типы в дереве, не ломая ядро.

## 5. ИИ как участник организации

### 5.1. Нулевой получатель

В WORK нет отдельного «модуля ИИ». ИИ — это **адресат по умолчанию**.

Любое действие пользователя в чате, у которого не указан явный человеческий получатель, **неявно адресовано ИИ**. ИИ — полноправный участник организации, который:

1. **Наблюдает** — видит поток операций через логи и журнал.
2. **Анализирует** — понимает, что произошло, в каком контексте и сколько это стоило.
3. **Планирует** — строит план достижения цели с помощью рассуждений.
4. **Действует** — вызывает скиллы, порождает новые операции и проводки.

### 5.2. Файлы как память ИИ

У ИИ **нет собственной долговременной памяти**. Файлы системы — это буквально память ИИ.

Неважно, какая модель «смотрит в систему» в момент ответов или действий. Любой из них должен быть подан контекст, необходимый для решения задачи с точки зрения пользователя и подразделения — буквально как целостное воспоминание, реконструированное из файлов, логов и журнала проводок.

Это делает WORK **модельно-независимым**: смена ИИ-провайдера (GigaChat → OpenAI → локальная модель) не требует миграции данных — вся память живёт в дереве.

### 5.3. Атрибуция действий ИИ

Действия ИИ-агентов должны быть атрибутированы именем модели (`sender: "GigaChat"`). Это нужно для:

- анализа качества моделей;
- сравнения результатов;
- восстановления происхождения изменений;
- доверия и аудита.

### 5.4. Скиллы — единица способности ИИ

**Скилл** — это зарегистрированная способность ИИ:

- имеет поля (что нужно для запуска);
- маршрутизируется (`skill-router` выбирает подходящий по эмбеддингам/ключевым словам);
- исполняется (`skill-manager` запускает выполнение);
- регистрируется как операция (файл `*.skill` + лог).

Набор скиллов эволюционирует — новые добавляются по мере развития потребностей организации.

Четыре направления ИИ — это не четыре подсистемы, а **четыре категории скиллов**:

1. **Коммуникации** — маршрутизация, дайджесты, нуджи, трансляция между форматами.
2. **Процессы** — восстановление, анализ, планирование, корректировка процессов.
3. **Ресурсы** — учёт, распределение, контроль ресурсов через единый журнал (см. раздел 8).
4. **Модификация** — самомодификация системы (см. раздел 10).

## 6. Чат как универсальный пульт

### 6.1. Не окно общения, а пульт управления

Чат — это **не «окно общения с ботом»**. Чат — это пульт управления организацией.

Через чат пользователь:

- пишет команды и сообщения → ИИ интерпретирует и действует;
- сохраняет файлы → они регистрируются как операции;
- создаёт объекты (`$class`) → структурированные операции;
- ведёт коммуникации → с людьми или с ИИ;
- регистрирует проводки → финансовые и количественные факты.

Один интерфейс, один канал, одна модель операций. Нет отдельных «форм создания задач», «экранов CRM», «мастеров workflow» — всё это проекции чата, которые ИИ и UI строят динамически под контекст.

### 6.2. Чат как представление событий

Чат не является отдельной базой сообщений. Он является **представлением над журналом событий класса**.

Когда пользователь пишет сообщение, оно сохраняется как `message.txt`. Ответ ИИ сохраняется как `response.md`. Ошибка — как `error.txt`. Всё проходит тот же путь: файл → history → log → отображение в chat.

Поэтому чат, аудит, RAG-память и процессная история сходятся в одну модель.

### 6.3. Звонки

Звонки — второй базовый интерфейс взаимодействия. Они работают рядом с чатом и также оставляют файлово-событийный след: приглашение, факт коммуникации, контекст, участники, результат.

## 7. Организационная модель

### 7.1. Иерархия класс

Организация моделируется иерархией класс (`$class`). Каждый узел дерева — подразделение, проект, рабочая точка.

На каждом уровне:

- **один руководитель** — задаёт вектор цели для этого узла;
- **один или несколько исполнителей** — действуют внутри, порождают операции.

### 7.2. Вектор цели

**Цель** — это не поле в конфиге. Цель — это **операция**.

Руководитель пишет промпт в чат → ИИ распознаёт его как скилл «формирование цели» → создаётся артефакт цели (файл) → он попадает в общий поток операций → ИИ в дальнейшем опирается на него при корректировке процессов.

Разница между целью и обычной операцией лишь в том, что ИИ трактует этот класс файлов как «директиву вектора»: опирается на него при планировании и оценке последующих операций.

### 7.3. Согласование целей снизу-вверх

Цель нижестоящего узла должна быть синхронизирована и согласована с целью вышестоящего.

Это не жёсткая иерархия прав, а **семантическая согласованность**, которую оценивает ИИ: «не противоречит ли цель отдела X миссии компании Y?». При выявлении рассогласования ИИ предлагает корректировку.

### 7.4. Права и роли

Права администратора проваливаются вниз по иерархии до тех пор, пока на промежуточном уровне не назначен свой администратор.

Назначенные пользователи-исполнители ведут деятельность и оставляют «хлебные крошки» — файлы, с которыми они работают.

Зона руководителя — типизирующие папки и наследуемый слой: через них он переопределяет `class.js`, handlers, методы. На практике это должен делать не сам руководитель вручную, а его ИИ-агент с соответствующими правами.

## 8. Ресурсы и единый журнал операций

### 8.1. Проблема факториального взрыва

Управление ресурсами организации (материалы, деньги, время, оборудование, люди) традиционно страдает от комбинаторного взрыва алгоритмов.

Если пытаться программировать «отдельный алгоритм под каждый тип операции», то при `n` типах операций количество возможных способов их взаимодействия равно `(n-1)!`. Реальное количество различных операционных моделей даже у небольшой организации превышает 1000. Факториал от этого числа — абсолютно нереальное количество кода для описания и поддержки.

Прямой алгоритмический анализ «сколько у нас материалов после всех этих операций» упирается в эту стену.

### 8.2. Единый журнал как математическая лазейка

**Единый журнал операций** (`register/`) — это математическая лазейка, обходящая факториальный взрыв.

Любая операция (любого типа!) **расщепляется на двойную запись**: дебетовую часть на один счёт + кредитовую часть на другой. Делая проводку, мы преобразуем **абсолютно любое количество операционных моделей к единой модели данных**.

Структура данных любой проводки практически одинакова:

- **операция** — ссылка на породивший артефакт/событие;
- **время** — точка во времени;
- **сумма** — оценочная стоимость (например, в рублях);
- **количество** — количественная мера;
- **корреспонденция** — связанный счёт (вторая сторона двойной записи);
- **аналитики** — ссылки на объекты справочников или операций (контрагент, договор, проект, подразделение, сотрудник...).

За исключением характерных для каждого ресурса аналитик, все счета описываются одной и той же моделью. Факториал `(n-1)!` схлопывается до **1**: достаточно **единого алгоритма** для анализа неограниченного количества операционных моделей.

### 8.3. План счетов как иерархия класс

Журнал реализован как дерево класс-счетов, подобное плану счетов бухгалтерского учёта, но значительно расширенное для учёта **всех значимых ресурсов организации** (не только финансов).

Пример структуры (уже частично существует в дереве):

```
register/                     — корневой журнал (баланс организации)
├── $register/                — метапапка журнала
├── 10/                       — счёт 10 «Материалы»
│   ├── $register/            — метапапка счёта
│   ├── 10.1/                 — счёт 10.1 «Сырьё»
│   │   └── $register/
│   └── 10.2/                 — счёт 10.2 «Запчасти»
│       └── $register/
├── 20/                       — счёт 20 «Основное производство»
├── 50/                       — счёт 50 «Касса»
├── 60/                       — счёт 60 «Расчёты с поставщиками»
└── ...                       — любой ресурс организации
```

Проводки нижестоящих счетов **агрегируются на вышестоящие** и так до уровня журнала. На верхнем уровне формируется **полный баланс организации** из всех записей во всех счетах.

### 8.4. Проводка как операция

Каждая рабочая группа или пользователь может видеть и добавлять записи (проводки) в виде файлов `.reg` сразу в **два счёта** (двойная запись: дебет-кредит), регистрируя:

- **количественный факт** — сколько единиц ресурса переместилось;
- **оценочный факт** — сколько это стоило (в рублях или иной валюте);
- **точку во времени** — когда это произошло;
- **набор аналитических признаков** — ссылки на объекты справочников или операций.

Проводиться в журнале могут:

- **факты операций** — оценочная себестоимость (отправлено 10 писем, сколько это стоило; сотрудник потратил 4 часа, оценить трудозатраты);
- **сами операции** — платёж, накладная, перемещение материалов, начисление зарплаты, списание, приход — любое фактическое или планируемое перемещение ресурсов.

### 8.5. Баланс и видимость

На верхнем уровне журнала формируется **баланс организации** — агрегат всех проводок по всем счетам.

Для каждого пользователя или подразделения доступны только соответствующие их правам счета. Это формирует **целостный прикладной баланс** для каждой точки ответственности: руководитель отдела видит баланс своего отдела, директор — баланс всей организации, бухгалтер — финансовый срез.

### 8.6. Связь с целями: от вектора к точным метрикам

Логи операций (раздел 4) дают ИИ **вектор** понимания — качественное направление («проект идёт хорошо/плохо»).

Единый журнал даёт ИИ **точные метрики** — количественные и финансовые показатели с высокой вычислительной эффективностью:

- сальдо (остаток) на счёте на момент времени;
- обороты (приход/расход) за период;
- агрегаты (sum, count, avg, min, max) по фильтрам времени и аналитик;
- цепочки корреспонденций (откуда → куда → зачем).

Журнал превращается в ещё одно классе фактов и связей с точными финансовыми и количественными показателями. На него ИИ опирается для управления достижением **точных целей** (а не просто векторов): «бюджет проекта X не должен превышать Y», «остаток материалов на складе Z должен быть не менее N».

### 8.7. Связь с кодом

В дереве уже существует:

- `register/` — корневой журнал (`$register`, label: «Журнал проводок»);
- `register/10/` — счёт 10 «Материалы» (`$register`, label: «10.00 Материалы»).

Предстоит реализовать:

- класс `$register` (или методы на `$class`) для расчёта сальдо/оборотов/балансов;
- формат файла проводки `.reg` (JSON с полями `time, sum, quantity, dt, ct, analytics`);
- агрегацию проводок снизу-вверх по иерархии счетов;
- API запросов к журналу с фильтрами и агрегаторами;
- связь проводок с операциями-артефактами через аналитики.

## 9. Границы системы и сеть

### 9.1. Целостный организм

WORK — не закрытая вселенная, а **целостный организм**:

- **чувствует** всё, что происходит внутри (полная регистрация операций и проводок);
- **видит** всё, что необходимо для достижения целей снаружи (через интеграции).

### 9.2. Сенсоры и акторы

Способов взаимодействия с внешним миром может быть огромное множество (SMTP/IMAP, Telegram, телефония, 1С, банковские API, CRM, IoT). Но все они заканчиваются одним: **сохранением файлов внутри системы**.

- **Сенсоры** — внешнее событие → операция-файл (письмо пришло → `inbox.eml`; звонок → запись звонка; банковская выписка → `.reg` проводки).
- **Акторы** — операция-файл → воздействие наружу (отправка письма → `outbox.eml`; вызов API 1С → `request.json`; платёж → банковский ордер).

Граница между WORK и внешним миром проходит по **факту регистрации**: как только внешний сигнал стал файлом в дереве, он стал частью памяти организации.

### 9.3. WORK как узел сети

WORK — один из узлов сети, в которой участники могут бесшовно взаимодействовать, открывая друг другу точки взаимодействия.

Например: поставщик сырья может шарить для своих клиентов точку в отделе продаж. Это означает, что класса разных организаций могут иметь **общие точки монтирования** — узлы дерева, видимые нескольким сторонам.

Это превращает WORK из одиночной инсталляции в **федерацию организаций**, связанных общими рабочими точками.

## 10. Самомодификация

### 10.1. WORK как живой организм

WORK — не застывшее приложение, а **саморазвивающаяся система**. Если организация меняется (новые процессы, документы, роли, интеграции), то и система управления должна меняться вместе с ней.

Четвёртое направление ИИ — **дальнейшая модификация системы** — означает, что ИИ (или человек через ИИ) должен уметь изменять сам WORK: создавать новые классы, методы, свойства, обработчики.

### 10.2. Уровни доверия

Самомодификация с человеческим контролем — это безопасно. Бесконтрольная — опасна. Поэтому вводятся **уровни доверия**:

1. **Скиллы** — ИИ создаёт новые файлы в `/$skill/...` (новые способности), не трогая ядро.
2. **Классы** — ИИ создаёт новые `$class` типы объектов, расширяя онтологию организации.
3. **Обработчики** — ИИ создаёт новые `~/handlers/methods/...`, расширяя поведение.
4. **Ядро** — ИИ редактирует код в `sources/` (с высшим уровнем одобрения).

### 10.3. Архитектура Cline-подобного контроля

Аналогия с ИИ-ассистентом в VS Code (Cline) — точная и сильная. WORK-ИИ должен:

- **видеть** всю кодовую базу `sources/` (классы, методы, свойства);
- **понимать** архитектуру (зависимости, что сломает изменение);
- **предлагать** модификацию (план → подтверждение → реализация);
- **применять** изменения в файлах дерева (`class.js`, handlers, triggers);
- **проверять** результат (тесты, инспекция состояния).

Если ИИ распознает запрос на изменение системы, он не применяет изменение сразу. Сначала он присылает руководителю план и/или дифф на подтверждение. После подтверждения ИИ-агент изменяет типизирующие папки, `class.js`, handlers — оставляя атрибутированный след.

### 10.4. Что для этого нужно

- **Метаданные доверия** — markings «этот файл может править ИИ», «требует approval», «только для чтения».
- **Инспекция кодовой базы** — API «дай определения всех методов класса X», «покажи зависимости».
- **Безопасный hot-reload** — после правки кода перезапуск или динамическая замена модуля.
- **Rollback** — откат к предыдущей версии артефакта (history уже есть, применение к коду — предстоит).

Соответствующие механизмы должны быть встроены во все классы в `sources/server`.

## 11. Файлово-объектное ядро

### 11.1. Три типа сущностей

Система состоит из сущностей трех типов:

- **`$folder`** — обычная папка.
- **`$file`** — файл.
- **`$class`** — особая папка-классе.

классе определяется наличием внутри папки ровно одной метапапки, имя которой начинается с `$`. Имя этой метапапки — тип класса.

### 11.2. Наследование и `class.js`

По цепочке типов сверху вниз «стекает» наследуемая информация: файлы, папки, handlers, настройки, представления и поведение.

Особый случай — `class.js`. Эти файлы сливаются слой за слоем как код через AST-merge (`babel-merge.js`). Это позволяет хранить на разных уровнях не полную копию объекта, а локальное уточнение: базовые свойства, настройки, поля, права, вычисляемые свойства, методы.

### 11.3. Три функциональных блока

1. **Файлово-объектное ядро** — базовая модель: `$folder`, `$file`, `$class`, `$register`. Хранение, типизация, наследование, метапапки, сборка `class.js`, история, логи, проводки, пути.
2. **Системный слой исполнения** — права, handlers, methods, pages, preview, серверные API, события сохранения, файловые обработчики, RAG-индексация, интеграции.
3. **Продуктовый слой** — Explorer, дерево, чат, звонки, офисные функции, формы, управление пользователями, ИИ-помощник, автопрограммирование.

## 12. RAG и память организации

RAG строится на файлах, логах и проводках деятельности. ИИ использует эту память, чтобы понимать процессы, последовательность, исполнителей, контекст подразделений, накопленные материалы и финансовые показатели.

RAG работает по дереву:

- внутри всех папок релевантного класса;
- вверх по организационной иерархии;
- с учётом локального и вышестоящего контекста.

Граница видимости опирается на права текущего пользователя.

## 14. Рабочая формула

WORK — это платформа для управления организационной деятельностью через файлово-событийную модель с глубокой интеграцией ИИ.

Организация представлена иерархией класс. Участники создают операции через чат, звонки и офисные действия. Каждая операция регистрируется как файл + лог в трёх адресатах. Ресурсы учитываются через единый журнал двойной записи, который схлопывает всё разнообразие операционных моделей к единой структуре. Логи становятся источником правды о прошлом, журнал — источником точных метрик, файлы — долговременной памятью ИИ. ИИ-участник наблюдает, анализирует, планирует и действует через скиллы. Руководители задают вектор цели. ИИ согласовывает процессы с этим вектором и достигает точных целей, опираясь на метрики журнала. Система развивает саму себя по мере роста организации.

## 16. Внутреннее устройство Reactor и `$public`

### 16.1. Reactor — реактивная база всех сущностей

Все сущности WORK (`$item`, `$folder`, `$file`, `$class`, `$user`) наследуются от `Reactor` (из `oda/reactor.js`).

**Ключевые механизмы:**

- **`target[R].props`** — единый реестр всех реактивных свойств экземпляра. Формируется один раз при первом обращении через статический геттер `Reactor[R]`:
  1. `proto2props(this.prototype)` — разворачивает свойства прототипа (включая `$public`) в реактивные дескрипторы.
  2. `join_props(parent.props, props)` — объединяет с реестром родителя (рекурсивно по цепочке прототипов).
  3. `Object.defineProperties(this.prototype, props)` — устанавливает реактивные геттеры/сеттеры.

- **`set DATA(data)`** — добавляет свойства из `class.js`:
  1. `proto2props(data)` — разворачивает объект из `class.js` в реактивные дескрипторы.
  2. `join_props(this[R].props, props)` — объединяет с существующим реестром.
  3. `Object.defineProperties(this, props)` — устанавливает на экземпляр.

- **`Reactor.get(target, key)`** — чтение реактивного свойства: проверяет кэш, вызывает оригинальный геттер, применяет конвертер типа, активирует вложенные объекты, инвалидирует зависимости.

- **`Reactor.set(target, key, value)`** — запись: конвертация типа, активация, вызов сеттера, инвалидация зависимостей.

### 16.2. `$public` — не модификатор доступа, а механизм сериализации

`$public` — это **не** «публичный vs приватный». Это механизм определения свойств, которые **сериализуются и передаются клиенту**.

**Как работает:**

1. Каждый класс определяет `get $public()`, возвращающий объект со свойствами (геттеры, значения).

2. `proto2props` рекурсивно разворачивает `$public` — каждое свойство становится реактивным свойством прототипа с флагом `$public: true`.

3. `join_props` объединяет `$public` по цепочке наследования:
   - `$item.$public` → `{id, name, label, type}`
   - `$folder.$public` → добавляет `{path, isInherit, isCustom}`
   - `$class.$public` → переопределяет `{isCustom}`, добавляет `{icon}`
   - `$user.$public` → переопределяет `{icon}`

4. `target[R].publics` — геттер, возвращающий массив имён свойств с флагом `$public`. Используется в `toJSON()` — только `$public`-свойства попадают в JSON, отправляемый клиенту.

**Следствие:** свойства, не входящие в `$public`, остаются реактивными (доступны через `this`), но не сериализуются. Клиент добирает их лениво через серверные методы.

### 16.3. `class.js` и наследование поведения

`class.js` — файл с `export default { ... }`. Через `importScript()` он импортируется как модуль, затем merge через `mergeScripts` (Babel AST) по цепочке наследования.

Результат — объединённый объект, который применяется к экземпляру через `set DATA()` → `join_props`. Свойства из `class.js` попадают в тот же `target[R].props`, но не автоматически `$public` — это зависит от определения в самом `class.js`.

### 16.4. Практическое следствие для инспекции

Для инспекции кодовой базы (метод `get_schema`) не нужно обходить прототипы вручную:
- `this[R].props` — уже содержит все реактивные свойства (JS-прототипы + `$public` + `class.js`).
- `this[R].publics` — уже содержит список сериализуемых свойств.
- `Object.getOwnPropertyNames(this.constructor.prototype)` — методы (функции).
