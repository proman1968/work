# Часть A. Документация

Стандарт системы документирования. Описывает принципы, структуру и механику ведения документации на каждом уровне иерархии.

---

## 1. Принципы

### 1.1. Иерархичность

Навигация по документации осуществляется через структуру папок. Путь к файлу сам по себе несёт контекст. Явные ссылки на родительские или дочерние readme не нужны — иерархия директорий обеспечивает связность.

### 1.1.1. Ссылки на внутренние файлы WORK

Когда всё же нужна кликабельная ссылка на файл внутри work (не «сырой» путь для Git), использовать только WORK-формат формы:

```markdown
[`sources/core.js`](/sources/core.js/~/handlers/pages/form/)
```

Шаблон: `/абсолютный/путь/к/файлу/~/handlers/pages/form/`.

- Относительные href вида `../../sources/page.html` для навигации в UI не использовать — в форме md они резолвятся относительно URL handler’а и ломаются.
- Внешние `http(s):` / `mailto:` / якоря `#` — как обычно.
- Viewer при показе дополнительно превращает подходящие пути в бэктиках и простые markdown-ссылки в WORK-формат; при ручном написании всё равно предпочитать явный WORK-href.

### 1.2. Двухуровневость

Каждый документ отвечает одновременно для двух потребителей:

- **Разработчик** (человек или ИИ-агент) — понимает техническое устройство для развития и модификации системы.
- **ИИ-управляющий процессами** — понимает прикладное назначение для управления задачами, процессами и ресурсами организации.

Оба измерения описываются в одном документе, без дублирования.

### 1.3. Минимализм

Документация не дублирует код. `readme.md` подробно описывает устройство и возможности, но не пересказывает реализацию. `progress.md` фиксирует факт + причину, а не пересказывает изменения в коде. Если код самодокументирован — документация описывает зачем и как это работает концептуально.

### 1.4. Актуальность

Документация обновляется вместе с кодом. Устаревший документ хуже отсутствующего. Раздел «Состояние» в `readme.md` и раздел «Последние изменения» в `progress.md` — индикаторы актуальности.

### 1.5. Глобальные механизмы — в корне

Глобальные механизмы (наследование, `~`, `$class`, `$public`, реактивность) описываются один раз в корневом `readme.md`. Локальные документы не повторяют их, а опираются на корневой контекст через иерархию.

### 1.6. Текущее → history (системный принцип)

Основной принцип WORK: **текущий файл перезаписывается целиком; каждое значимое сохранение даёт полную неизменную копию в history**. Его реализует цепочка **`save_file` → `history/<date>/время.uid.ext` → log** (и далее лента / RAG).

Инварианты:

1. **Текущее** — одно актуальное состояние; каждый save **заменяет** файл целиком (не «дописывание» к прошлой версии).
2. **History** — полная копия **этой** версии на момент save; снимок не редактируют.
3. **Разные снимки** — разное содержимое (как разные версии `$file`). Не копировать раздутый лог с повторяющимися строками из предыдущего save.
4. **Дата** — в папке `history/YYYY-MM-DD/`; **время** — в имени снимка.
5. На момент снимка: **текущий файл ≡ этот снимок**.

Изоморф в разработке:

| Слой | Текущее | History |
|------|---------|---------|
| Платформа `$file` | файл на диске | `history/<date>/<ms>.<uid>.ext` |
| VSCode / агент | `progress.md` | `.progress.md/history/<date>/<ms>.<user>.md` |

`progress.md`: каждый значимый апдейт — **новый** файл; в «Последние изменения» только события **этого** save. Предыдущие версии — только в history. Не дописывать буллеты к старому progress.

### 1.6.1. Артефакты задачи — один файл

ИИ и люди пишут артефакт задачи (презентация, отчёт, заметка) в **одно конечное имя** (`presentation.html`, `readme.md`). Каждый `save_file` / сохранение **перезаписывает** тот же файл — платформа сама копит history по §1.6. Промежуточные имена (`*.struct.*`, `*.draft.*`, отдельный outline рядом с финалом) запрещены: они рвут непрерывную историю результата.

### 1.7. Машиночитаемость

Структура разделов единая и предсказуемая. Это позволяет ИИ эффективно навигировать по дереву знаний, понимать контекст любой точки и использовать инструменты системы. Отклонения от стандарта недопустимы.

### 1.8. Человекоориентированность

`readme.md` хорошо оформлен и понятен. Markdown с заголовками, списками, таблицами. Не сухой протокол, а понятный рассказ о точке системы. Документ пишется для людей, а не только для машин.

### 1.9. Именование классов

Папка **класса** (`create` с `type: '$class'`) именуется **целиком ЗАГЛАВНЫМИ буквами** (`MARKET`, `SYS`, `BASE`, `СТАРТ`). Алфавит любой (латиница или кириллица). `label` в `class.js` — произвольный.

Обычные системные папки (`oda`, `sources`, `docs`, `lib`, …) — не заглавными: так в корне сразу видно класс. Инстансы других типов (`$paas` с DNS-id и т.п.) под это правило не подпадают.

## 3. Структура `readme.md`

Каждый `readme.md` содержит 6 разделов в указанном порядке:

### 3.1. Что это

Назначение элемента. Одно-два предложения. Техническое и прикладное измерение.

### 3.2. Зачем это нужно

Роль в архитектуре и в управлении. Какую проблему решает. Зачем существует.

### 3.3. Как это работает

Принципы и ключевые механизмы. Концептуальное описание, не код. Доступные функции и инструменты — для ИИ-управляющего.

### 3.4. Из чего это состоит

Перечень содержимого: файлы, подпапки, классы, методы. Каждый элемент описан в двух измерениях — техническое имя + прикладное назначение в скобках.

Пример:

```
- `folder.js` — класс `$folder`: дерево элементов, `children`, `get_item` (управление деревом папок, навигация, доступ к дочерним элементам)
- `class.js` — класс `$class`: merge/diff, logs, metadata (наследование поведения, журнализация операций, доступ к истории)
```

### 3.5. В каком это состоянии

Честный статус. Что реализовано, что работает, что в процессе, что не работает. Метки:

- ✅ работает
- 🔧 в разработке
- 🧪 прототип / эксперимент
- ❌ не реализовано
- ⚠️ устарело

### 3.6. Дальнейшие планы

Направления развития. Открытые вопросы. Что планируется сделать.

## 4. Структура `progress.md`

Применение системного принципа §1.6 (`save_file → history → log`) к журналу разработки.

```markdown
# Прогресс: <название>

## Последние изменения
- [ЧЧ:ММ] Сделано X — причина/решение

Только события **этого** save. Только `[ЧЧ:ММ]` (не дата).
Дата — в папке `.progress.md/history/YYYY-MM-DD/`.

## В работе
- Текущая задача (актуально на этот момент)

## Ключевые решения
- Решение: выбор A вместо B. Причина: ... (актуально на этот момент)

## Блокеры / Открытые вопросы
- Проблема X — ждём Y
```

### 4.1. Принципы `progress.md`

1. **Локальность** — на каждом уровне свой progress.
2. **Каждый save — новый файл** — перезапись целиком; «Последние изменения» = только буллеты **этого** действия. Не дописывать к предыдущему progress.
3. **Снимки уникальны** — разные файлы history не содержат один и тот же набор буллетов изменений (как разные версии `$file`).
4. **Краткость** — одно-два предложения на запись. Факт + причина.
5. **Решения важнее действий** — «почему» ценнее перечисления файлов.
6. **Архив = history** — прошлые версии только в `.progress.md/history/`. Не секция `## Архив` внутри `progress.md`.

## 5. Механика `progress.md` (изоморф §1.6)

При каждом значимом действии ИИ-агента:

1. **Собрать новый** `progress.md` с нуля: в «Последние изменения» — только события **этого** save (`[ЧЧ:ММ]`). Разделы «В работе» / «Ключевые решения» / «Блокеры» — актуальное состояние **сейчас**, не архив прошлых буллетов изменений.
2. **Записать** файл целиком (перезапись).
3. **Скопировать as-is** в `.progress.md/history/YYYY-MM-DD/TIMESTAMP.USER.md`.

Инвариант: **`progress.md` ≡ последний снимок**. Запрещено: merge/дописывание строк из предыдущего `progress.md`; копирование раздутого лога; правка старых снимков; `[ДД.ММ.ГГГГ]` в строках.

Чеклист перед записью: открыть мысленно пустой шаблон — не читать старый progress чтобы «добавить сверху».

Формат имени снимка: `TIMESTAMP.USER.md` внутри `history/YYYY-MM-DD/`

- `YYYY-MM-DD` — дата (папка, как в WORK)
- `TIMESTAMP` — время в миллисекундах (как в WORK)
- `USER` — имя агента / разработчика

Пример: `.progress.md/history/2026-07-19/178445411268.cline.md`

### 5.1. Структура папки

```
<папка>/
├── readme.md                                ← описание
├── progress.md                              ← только последний save (перезаписывается)
└── .progress.md/
    └── history/
        └── 2026-07-19/
            ├── 178445411268.cline.md        ← версия A (свои буллеты)
            ├── 178445500000.cline.md        ← версия B (другие буллеты, без копипаста A)
            └── ...
```

### 5.2. Git

`.progress.md/history/` можно добавить в `.gitignore` (как локальный аналог `history/` в WORK). Решение за пользователем.---

# Часть B. Декларативный UI (главное правило кода)

Чтобы ИИ и люди читали и меняли интерфейсы **без костылей**. Реактор рисует от данных; мутация JSON → UI.

## B.1. Паттерн

1. Источник правды — один объект `data` (JSON файла или блок ленты).
2. Списки — геттер от `data`, бинд `:items` / `:ribbon`.
3. Элемент списка — **`:data="$for.item"`**, не `~props="$for.item"`.
4. View читает поля **геттерами от `data`**.
5. Базовый компонент объявляет `data`; наследники — `extends`.

```html
<!-- правильно -->
<div ~is="tag($for.item)" :data="$for.item"></div>
<item-node :$item></item-node>

<!-- неправильно -->
<div ~is="..." ~props="$for.item"></div>
```

```js
ODA({ is: 'microchat-view', data: null });

ODA({
    is: 'microchat-view-file',
    extends: 'microchat-view',
    get path() { return this.data?.path; },
    get $item() {
        return this.path && WORK.get_item(this.path);
    },
});
```

Пример оболочки:

```js
data: null, // JSON task.ai
get items() { return this.data?.ribbon || []; },
// template: <microchat-ribbon :items></microchat-ribbon>
```

## B.2. Запрещено

- Размазывать блок JSON по пропсам через `~props` / `assignProps`.
- Дублировать поля на view (`path: ''` рядом с `data.path`).
- Hydrate/`$file` на ленте, если view сам резолвит `WORK.get_item(this.path)`.
- Сжимать код удалением переносов или CSS «в одну строку» ради метрики строк.

## B.3. Отладка

Смотреть `$0.data` и геттеры. Если `path === undefined` при живом блоке в JSON — сломан бинд `:data`, а не «файл не найден».
