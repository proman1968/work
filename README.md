# WORK — платформа управления деятельностью организаций

Платформа для управления деятельностью организаций через файлово-событийную модель с глубокой интеграцией ИИ. Фундаментальная архитектурная декларация — в [`whitepaper.md`](/whitepaper.md/~/handlers/pages/form/). Стандарт документации — в [`rules.md`](/rules.md/~/handlers/pages/form/). История работы — в [`progress.md`](/progress.md/~/handlers/pages/form/).

---

## 1. Что это

WORK — файлово-объектная платформа, где **всё есть файл**, а организация моделируется иерархией `$class`. Каждая операция регистрируется как файл-артефакт + лог-факт. ИИ — полноправный участник, который наблюдает, анализирует, планирует и действует через скиллы.

## 2. Зачем это нужно

WORK решает проблему управления организационной деятельностью единым способом: вместо разрозненных CRM, ERP, таск-трекеров и чатов — одна файлово-событийная модель, где чат, аудит, RAG-память и процессная история сходятся в одну точку. ИИ-управляющий понимает состояние организации через файлы и логи, планирует действия и достигает целей.

## 3. Как это работает

### 3.1. Реактивная модель

Все сущности WORK (`$item`, `$folder`, `$file`, `$class`, `$user`) наследуются от `Reactor` (из `oda/reactor.js` / `sources/reactor.js`).

- **`target[R].props`** — единый реестр всех реактивных свойств экземпляра. Формируется один раз при первом обращении через статический геттер `Reactor[R]`.
- **`set DATA(data)`** — добавляет свойства из `class.js` в реестр.
- **`Reactor.get(target, key)`** — чтение: проверяет кэш, вызывает оригинальный геттер, конвертирует тип, инвалидирует зависимости.
- **`Reactor.set(target, key, value)`** — запись: конвертация, вызов сеттера, инвалидация зависимостей.
- Геттеры + события `changed`/`reset()` для автообновления. Не поллинг.

### 3.2. `$public` — механизм сериализации

`$public` — это **не** «публичный vs приватный». Это механизм определения свойств, которые **сериализуются и передаются клиенту**.

- Каждый класс определяет `get $public()`, возвращающий объект со свойствами.
- `proto2props` разворачивает `$public` в реактивные свойства с флагом `$public: true`.
- `target[R].publics` — массив имён сериализуемых свойств. Используется в `toJSON()`.
- Свойства вне `$public` остаются реактивными, но не сериализуются. Клиент добирает их лениво через серверные методы.

### 3.3. Три типа сущностей

| Тип | Назначение |
|---|---|
| `$folder` | Обычная папка |
| `$file` | Файл (наследует `$folder`) |
| `$class` | Особая папка-класс (содержит ровно одну метапапку с именем, начинающимся с `$`) |

### 3.4. Наследование и `~` (tilde)

По цепочке типов сверху вниз «стекает» наследуемая информация: файлы, папки, handlers, настройки, представления и поведение.

- **`~` (tilde)** — механизм поиска типизирующих файлов по цепочке типов. `collect_tilde()` собирает файлы из метапапок вверх по иерархии.
- **`class.js`** — merge по слоям через AST-merge (`babel-merge.js`). Каждый слой — локальное уточнение, не полная копия.
- **`$server/$folder/`** — корневой типизатор. Его содержимое доступно каждому классу через `~`.

### 3.5. Операция → файл → лог

Любое действие, меняющее состояние организации, проходит один путь:

1. Инициатор (пользователь, ИИ, handler, сенсор) → действие.
2. Сохраняется файл-артефакт.
3. `$class` раскладывает файл по назначению (text, image, logs, js, …).
4. Создаётся копия в `history/<date>/время.uid.ext` — неизменный снимок.
5. Создаётся запись в `data.logs` — факт операции.
6. Запись попадает в ленту событий, RAG-индексацию, файловые обработчики.

### 3.6. Единый журнал (двойная запись)

Любая операция расщепляется на дебетовую + кредитовую части. План счетов — иерархия `$class`-счетов в `register/`. Баланс организации = агрегат всех проводок снизу-вверх. Факториал операционных моделей схлопывается до единой структуры.

### 3.7. ИИ как участник

ИИ — адресат по умолчанию (нулевой получатель). Файлы системы — долговременная память ИИ (модельно-независимость). Действия ИИ атрибутируются именем модели. Скиллы — единица способности, эволюционируют.

## 4. Из чего это состоит

### Движок

| Путь | Назначение |
|---|---|
| `sources/reactor.js` | `Reactor` — реактивная база всех сущностей (единый реестр свойств, геттеры/сеттеры, инвалидация) |
| `oda/oda.js` | ODA — визуальный фреймворк: декларативное компонентное построение UI, наследование, библиотеки компонентов |
| `sources/core.js` | `$item extends Reactor` — общий абстрактный базовый класс (среда-независимый код) |
| `sources/work.js` | Серверный entrypoint: создаёт `WORK = new $server()`, запускает HTTP/HTTPS, WebSocket, STUN |
| `sources/client.js` | Клиентский entrypoint: загружает ODA, клиентскую модель, регистрирует `WORK`, UI, браузерные сервисы |
| `run.mjs` | Запуск сервера (Node.js) |

### Серверная объектная модель (`sources/server/`)

| Файл | Назначение |
|---|---|
| `folder.js` | `$folder`: дерево элементов, `children`, `get_item`, tilde, manifest (управление деревом папок, навигация) |
| `class.js` | `$class`: class.js merge/diff, logs, secrets, metadata (наследование поведения, журнализация, конфигурация) |
| `file.js` | `$file`: load/save, history, edit_file, триггеры on_save (чтение/запись файлов, история версий) |
| `handler.js` | `$handler extends $class`: исполняемый элемент (запуск handlers) |
| `user.js` | `$user`: пользовательская сущность, online-статус (управление пользователями) |

### Системный типизатор (`$server/$folder/`)

| Путь | Назначение |
|---|---|
| `$file/` | Тип файлов и подтипы по расширениям: `$ai`, `$prompt`, `$txt`, `$md`, `$eml`, `$image`, … (обработка файлов по типам) |
| `$class/` | Тип класс и подтипы: `$class`, `$device`, `$structure` (типизация объектов) |
| `$handler/` | Тип обработчиков: `$task`, `$trigger` (реакции на события) |
| `handlers/` | Серверные и клиентские обработчики: `methods/`, `pages/` (формы, страницы, API) |
| `lib/` | Библиотеки прикладных компонентов: `chat-item`, `tree`, `explorer`, `security`, … (переиспользуемые модули) |

### Прикладные зоны

| Путь | Назначение |
|---|---|
| `SYS/` | Системный слой: Billing (кошелёк, ЮKassa), Licenses (self-signed `.lic`). PROTECTED-зона |
| `Offerings/` | Продуктовые услуги: базовый `$offering` (заявки) + PaaS (proposals, provision) |
| `models/` | AI-модели: GigaChat, z.ai и др. (подключение к внешним LLM) |
| `services/` | Внешние сервисы: SearXNG, Weather, ArgoCD (интеграции) |
| `skills/` | Скиллы ИИ: зарегистрированные способности (маршрутизация, исполнение) |
| `register/` | Единый журнал проводок (двойная запись, баланс организации) |
| `nodes/` | Узлы сети WORK (федерация организаций) |
| `base/` | Базы данных и справочники |
| `users/` | Пользователи |
| `tests/` | Тесты |
| `torus/` | Нейросетевой фреймворк (экспериментальный) |
